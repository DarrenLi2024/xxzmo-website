## 修复后台管理页面和前端 500 错误

### 问题根因
Vercel 生产环境的数据库连接字符串指向了旧的 **Neon 数据库**，而本地 `.env` 配置的是新的 **Supabase 数据库**。两者完全不同：
- Neon：有 64 篇文章数据，但缺少新 schema（AiWorkflowRun、AiArtifact 等表，以及 Article.aiStatus 等列）
- Supabase：有完整 schema，但数据为空

这导致：
- 文章列表页（/jigu、/chuli）返回 200 但内容为空（Prisma 查询在 try-catch 中失败，返回空数组）
- 首页可能 500（home-queries.ts 没有 try-catch）
- 管理后台 API 500（数据库 schema 不匹配）

### 修复步骤
1. **诊断**：部署调试端点，发现生产环境连接的是 `ep-polished-smoke-...neon.tech`（Neon），不是 Supabase
2. **修复环境变量**：更新 Vercel 的 `POSTGRES_PRISMA_URL` 和 `DATABASE_URL` 为 Supabase 连接字符串
3. **数据迁移**：从本地 `prisma/dev.db`（SQLite）导出数据，批量导入到 Supabase
   - Article: 64 条
   - Tag: 67 条
   - TagOnArticle: 66 条
   - Painting: 40 条
   - SiteConfig: 1 条
   - DailyQuote: 12 条
   - LlmProvider: 6 条
   - PinyinDict: 217 条
4. **添加错误处理**：给 `src/lib/home-queries.ts` 的所有查询函数添加 try-catch，避免数据库异常导致首页 500
5. **重新部署**：部署到 Vercel 生产环境

### 验证结果
- 首页 `/`：200 ✅
- `/jigu`：200 ✅
- `/chuli`：200 ✅
- 文章详情页：200 ✅

## 性能优化（下午补充）

### 问题
- 首页加载 9 秒，/jigu 3.8 秒，/api/health 4 秒
- 根因：Vercel（美国）与 Supabase（首尔）跨国网络延迟 + connection_limit=1 导致并发排队

### 优化措施
1. **ISR 缓存**：首页、文章列表、文章详情添加 `revalidate = 30`（30 秒缓存）
2. **连接池扩大**：`connection_limit` 从 1 改为 5
3. **查询优化**：文章列表 `take` 从 200 减到 60，使用 `select` 精确取字段
4. **首页查询并行化**：4 个查询改为 `Promise.all` 并行

### 验证结果
- 首页：9s → 0.57s（ISR 缓存命中）
- /jigu：3.8s → 0.23s（ISR 缓存命中）
- 所有 API 500 错误已修复（返回 401 或 200）
